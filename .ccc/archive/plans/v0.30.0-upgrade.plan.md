## Phase 2: CPU=0 误杀修复

### 目标
`_check_and_mark_hung()` 用 `ps -o %cpu=` 判断进程是否 hung。
macOS 上 `%cpu=` 是生命周期均值，运行 >5min 的 LLM 调用纯网络等待时 CPU 接近 0，被误杀。

### 改动清单

#### 2.1 ccc-engine.py `_check_and_mark_hung()` CPU 判定逻辑

**文件**：`scripts/ccc-engine.py`
**函数**：`_check_and_mark_hung`（~L2150）

**当前逻辑**（伪代码）：
```
if cpu > 0.0:
    continue  → 不标记 hung
# 若这里没有被 continue，就标记 hung
```

**改为**（伪代码）：
```
if cpu > 0.0:
    continue  → 不标记 hung（进程确实在工作）

# cpu <= 0.0 — 有可能是真的 hung，也有可能是纯网络等待
# 增加二次确认：检查最后文件修改时间
pid_dir = pids_dir / str(pid)
age_check = 120  # 2 分钟无新文件活动才认为 hung
try:
    latest_mtime = max(
        f.stat().st_mtime for f in pids_dir.iterdir()
        if f.name.startswith(f"{subid}.") and f.is_file()
    )
    file_age = (now.timestamp() - latest_mtime)
    if file_age < age_check:
        continue  → 最近 2 分钟有文件活动，不标记 hung
except (OSError, ValueError):
    pass  → 读不到 mtime，回退到 CPU 判定
```

**详细修改**：

找到以下代码块（约 L2168-2174）：

```python
try:
    cpu = float(result.stdout.strip())
except ValueError:
    engine_log(...)
    continue
if cpu > 0.0:
    continue
```

**在其后添加**（`if cpu > 0.0: continue` 的 else 分支）：

```python
# macOS ps -o %cpu= 返回的是生命周期均值。一个长期运行的进程
#（如 opencode 等待 LLM 调用）总 CPU 可能极低但没 hung。
# 二次确认：检查 pids 目录下的附属文件最后修改时间。
_latest = 0.0
try:
    for _pf in sorted(
        pids_dir.iterdir(), key=lambda x: x.stat().st_mtime, reverse=True
    ):
        if _pf.name.startswith(f"{subid}."):
            _latest = _pf.stat().st_mtime
            break
except OSError:
    pass
if _latest and (now.timestamp() - _latest) < 120:
    # 最近 2 分钟还有文件活动 → 不是 hung
    continue
```

### 禁止事项

- 不要简单把 `_HANG_CHECK_INTERVAL_SEC` 从 300 改成更大（治标不治本）
- 不要移除 `ps %cpu=` 检查（对 CPU 密集型进程仍然有效）
- 不要改动 patrol-v4.py 中的 `engine_is_running()`（它用 heartbeat 逻辑，与本函数不同）

## Phase 3: Verdict 门禁

### 目标
当前 FAIL/FALLBACK verdict 写入后，commit 已入库。没有回滚机制。
改为：Engine 下一个 tick 检测到 FAIL/FALLBACK verdict → git revert → 退回 planned。

### 改动清单

#### 3.1 ccc-engine.py 新增函数

**文件**：`scripts/ccc-engine.py`

在 `_verdict_is_timeout()` 函数附近新增（建议放在 L460 之后）：

```python
def _revert_task_commit(ws: Path, tid: str) -> bool:
    """Verdict FAIL 时回滚 task 的最后一个 commit。

    按 task_id 查找最近一个 commit（从 phases.json 取 commit hash → git revert），
    退回 planned 列，清理 active_tasks。

    Returns:
        True = 回滚成功；False = 失败（不阻断 engine 运行，仅日志）
    """
    # 1. 读 phases.json 取 commit hash
    phases_file = ws / ".ccc" / "phases" / f"{tid}.phases.json"
    if not phases_file.exists():
        return False
    try:
        phases_data = _load_phases(tid, ws)
        commits = [
            p.get("commit", "") for p in phases_data
            if p.get("commit") and p.get("commit") not in ("null", "None", "")
        ]
        if not commits:
            return False
        last_commit = commits[-1]
    except (OSError, json.JSONDecodeError) as exc:
        engine_log(f"[verdict-gate] {tid} 读 commit hash 失败: {exc}")
        return False

    # 2. git revert（使用 subprocess，走 _sanitized_env）
    try:
        import subprocess as _sp  # noqa: 模块级已有 subprocess

        result = _sp.run(
            ["git", "revert", "--no-edit", last_commit],
            capture_output=True, text=True, timeout=30,
            cwd=str(ws), env=_sanitized_env(),
        )
        if result.returncode != 0:
            engine_log(
                f"[verdict-gate] {tid} revert {last_commit[:12]} 失败: "
                f"{result.stderr[:200]}"
            )
            return False
        engine_log(
            f"[verdict-gate] {tid} 已 revert commit {last_commit[:12]}"
        )
        return True
    except Exception as exc:
        engine_log(f"[verdict-gate] {tid} revert 异常: {exc}")
        return False
```

### 4.2 子进程上限

**文件**：`scripts/ccc-engine.py`

**目标**：限制全局并发 opencode 子进程数。当前 3 tasks × 2 parallel phases = 6 processes，且只有 active_tasks 计数器没有真实进程计数。

**位置**：Engine 主循环中 `_try_launch_planned()` 等启动子进程的函数。

**方案**：新增全局计数器 `_GLOBAL_OPENCODE_COUNT`（模块级变量），每个 launch 前 +1，`.done` 标记后 -1。launch 前检查计数是否超限。

```python
# 模块顶部添加（约 L120 附近）
# v0.30.0: 全局 opencode 并发计数
_GLOBAL_OPENCODE_COUNT = 0
_GLOBAL_OPENCODE_MAX = 6  # ∑ (MAX_CONCURRENT × PHASE_PARALLEL_MAX_WORKERS)
```

**修改 `_exec_multi_phase`（或启动 phase 的函数）**：

launch 前：
```python
global _GLOBAL_OPENCODE_COUNT
if _GLOBAL_OPENCODE_COUNT >= _GLOBAL_OPENCODE_MAX:
    engine_log(
        f"[engine] 全局 opencode 已达上限 "
        f"({_GLOBAL_OPENCODE_COUNT}/{_GLOBAL_OPENCODE_MAX})，等待"
    )
    return False
_GLOBAL_OPENCODE_COUNT += 1
```

phase 完成回调（`.done` 检测处）：
```python
_GLOBAL_OPENCODE_COUNT = max(0, _GLOBAL_OPENCODE_COUNT - 1)
```

**注意**：`_GLOBAL_OPENCODE_COUNT` 是进程内变量，多 workspace 共享同一进程（Engine 单进程），所以有效。

**风险**：`try/finally` 确保任何 launch 失败路径下 `_GLOBAL_OPENCODE_COUNT` 能回退。

## 改动 1：核心 scheduler 脚本 + 红线 13 写入 + launchd 文档增强

### 做什么

把"任务表驱动循环"从 README/roadmap 落地成可执行脚本。`ccc-scheduler.sh` 从 `.ccc/queue/` 目录读任务(JSON 文件,每个代表一个待跑 CCC 任务),按优先级排序、串行执行、失败重试 N 次后挂起,启动前强制跑 watchdog。`ccc-task-done.sh` 是调度循环里的"完成回调"——单 phase 完成后 commit + 标记下一 phase + 把当前任务从 in_progress 移到 done,出错时把整个任务标 failed 并停止调度器。

同时把红线 13 写入 `references/red-lines.md`(roadmap v0.6 已声明但 red-lines.md 还没有),与 `scheduler-launchd.md` 文档配套:文档明确"启动前必须跑 `bash scripts/executor-watchdog.sh || exit 2`,失败立即停调度"——这是 v0.6 的硬安全约束,防止自动 fail-loop 死锁烧预算。

### 怎么做

**新增 `scripts/ccc-scheduler.sh` ~50 行**:
- bash 5.x + set -euo pipefail
- 参数:`--queue-dir <path>`(默认 `.ccc/queue/`)、`--max-retries N`(默认 3)、`--interval-sec N`(默认 30)、`--dry-run`、`--once`(单次跑不入循环)
- 启动顺序:先调用 `bash scripts/executor-watchdog.sh`——非零退出立即 exit(红线 13 强证据)
- 主循环:`for task_file in queue_dir/*.json` 按优先级排序,逐个跑 `ccc-task-done.sh <task>` 风格的处理逻辑
- 错误处理:单 task 重试 N 次 → 第 N+1 次失败 → 写 `.ccc/abnormal-reports/scheduler-failed-<task>-<date>.md` + exit
- 日志:写到 `~/.ccc-logs/scheduler-<date>.log`(路径可覆盖)
- dry-run 模式只打印要做的事,不真跑 Executor

**新增 `scripts/ccc-task-done.sh` ~30 行**:
- bash 5.x
- 参数:`<workspace> <task-id> [--skip-commit]`
- 流程:跑 `bash scripts/ccc-finish.sh <workspace> <task>` → 0 退出 → commit(若未 skip)→ 更新 `.ccc/state.md` 任务状态 → 解锁 queue 里下一 task

**修 `references/red-lines.md`** 追加红线 13 段:
- 在红线 12 后、红线违反处理流程前插入
- 完整结构:Why + 机制钩子(`ccc-scheduler.sh` 启动前 watchdog) + 触犯后果(Critical)

**修 `references/adapters/scheduler-launchd.md`** 66 → ~90 行:
- 加"⚠️ 红线 13 强约束"段:启动 plist 前必跑 watchdog
- plist 模板里 `ProgramArguments` 第一项改为 `bash scripts/ccc-scheduler.sh --once` 而非裸 `claude -p`
- 加 `<key>ExitTimeOut</key>` + `<key>ThrottleInterval</key>` 防高频触发
- 加卸载步骤示例(`launchctl bootout gui/$UID <plist>`)

### 验收

- `bash scripts/ccc-scheduler.sh --dry-run --once` 退出码 0,输出含"watchdog OK" + "queue: N tasks" + "would run: <task>" 三段,且 watchdog 真跑了(`grep executor-watchdog` 在日志里有输出)
- `bash scripts/ccc-scheduler.sh` 不带 watchdog 时(或手动让 watchdog 失败,如 `--hang-threshold 0` 触发 hang 检测)退出码非 0 + 立即 exit(无主循环跑起来)
- `bash scripts/ccc-task-done.sh <ws> <task>` 在 task 不存在时退出码 2 + 输出"ERROR: report not found"
- `bash scripts/ccc-task-done.sh` 不带参数时退出码 2 + 输出 usage
- `grep -E "^### 红线 13" references/red-lines.md` 命中,且红线 13 段含 "watchdog" 关键词
- `references/adapters/scheduler-launchd.md` 含 "红线 13" 字符串 + `ProgramArguments` 段引用 ccc-scheduler.sh
- `bash -n scripts/ccc-scheduler.sh` + `bash -n scripts/ccc-task-done.sh` 两脚本 SYNTAX OK
- 参考命令:`bash scripts/ccc-scheduler.sh --dry-run --once`、`bash scripts/ccc-task-done.sh . nonexistent-task`、`grep -E "^### 红线 13" references/red-lines.md`

## 改动 2：launchd plist + GitHub Actions 配套模板 + 任务表示例

### 做什么

提供 3 个"开箱即用"的 scheduler 模板,让用户 clone CCC 后 30 秒内能跑起来自动调度循环,不需自己写 plist 或 GitHub Actions workflow。模板与 Phase 1 的 `ccc-scheduler.sh` 配套——plist 启动的是 scheduler 脚本而非裸 `claude -p`,符合红线 13。

### 怎么做

**新增 `examples/scheduler/ccc-queue.plist` ~30 行**:
- macOS launchd plist 格式,Label 用 `com.ccc.scheduler`(取代之前的 `ccc.qxo.<task>` 占位符,通用化)
- `ProgramArguments` 第一项 `bash`、第二项 `ccc-scheduler.sh` 绝对路径、第三项 `--once`(配合 launchd 自身的 `StartInterval` 做循环,而非脚本内循环,更易调试)
- `StartInterval` 设为 300(5 分钟,与 roadmap v0.6 默认一致)
- 加 `RunAtLoad=false` + `StandardOutPath`/`StandardErrorPath` 指向 `~/.ccc-logs/`
- 注释说明:加载前需 `chmod +x scripts/ccc-scheduler.sh` + 修改 `<string>/Users/apple/program/CCC/scripts/ccc-scheduler.sh</string>` 为真实路径

**新增 `examples/scheduler/ccc-queue.yml` ~30 行**:
- GitHub Actions workflow,与 `scheduler-github-actions.md` 配套
- 触发器:`schedule.cron: '*/30 * * * *'`(30 分钟一次,plausible 演示频率)+ `workflow_dispatch` 手动触发
- Job 跑 `bash ~/program/CCC/scripts/ccc-scheduler.sh --once`
- env 注入 `ANTHROPIC_API_KEY` + `CCC_WORKSPACE`(默认 `.`)

**新增 `examples/scheduler/queue.example.json` ~10 行**:
- 任务表示例:`[{"task": "hello-ccc-demo", "priority": 1, "max_retries": 3}]`
- 含 2-3 个示例任务用于测试 scheduler 排序

### 验收

- `plutil -lint examples/scheduler/ccc-queue.plist` 输出 "OK"(macOS plist 语法校验)
- `python3 -c "import yaml; yaml.safe_load(open('examples/scheduler/ccc-queue.yml'))"` 不抛异常
- `python3 -c "import json; json.load(open('examples/scheduler/queue.example.json'))"` 不抛异常
- plist 含 `com.ccc.scheduler` Label + `ccc-scheduler.sh` 引用 + `StartInterval` 整数 300
- yml 含 `schedule.cron` + `ccc-scheduler.sh --once` 引用
- queue.example.json 是合法 JSON 数组且至少 1 条任务
- 参考命令:`plutil -lint examples/scheduler/ccc-queue.plist`、`python3 -c "import json; json.load(open('examples/scheduler/queue.example.json'))"`

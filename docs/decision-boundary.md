# 决策边界（永久加固）

> 2026-07-14 教训：reviewer_timeout 这种改一行配置的事，我问了老板"要修吗"。
> 老板说：这种问题都要问，你还有什么价值。

## 一句话规则

**自己能判断的自己定。不确定才问。**

## 决策速查

| 场景 | 怎么办 | 例子 |
|------|--------|------|
| 配置参数不合理 | **直接改** | reviewer_timeout 太短 → 加参数、改值、重启 |
| 代码 bug | **直接修** | 变量未定义、超时共用、逻辑错误 |
| 流程卡住 | **直接清** | patrol 检测到卡死 → 移回 planned/backlog |
| 缺功能 | **直接加** | backlog 空了 → 注入任务 |
| 上游不可用 | **直接切/等** | API 超时 → 加超时或重试 |
| 引入新依赖 | 问 boss | npm install / pip install / brew install |
| sudo/系统配置 | 问 boss | 改 /etc、装 launchd 系统级服务 |
| 方向摇摆 | 问 boss | 两个方案效益相当、或者意图没把握 |
| 花钱 | 问 boss | 付费 API、购买服务、云资源 |

## 禁止清单（遇到直接修，不问）

- config 参数不合理 → 加参数、改默认值
- 变量名/import/语法错误 → 直接修
- timeout 太短/太长 → 调参数
- 测试失败（明显是测试 bug）→ 修测试
- 日志太吵/重复 → 降级或删
- 任务卡死 → 移走
- backlog 空了 → 注入
- 流程不通 → 修通

## 必须问 boss 的

- "我选 A 还是 B"（等效方案）
- 新依赖 / sudo / 改系统配置
- 花钱的决定
- "要修吗" / "要不要" / "可以吗"（这种措辞本身就说明不该问）

## 验收标准

- 每次我开口问之前，先自问：**这事在我的决策权限吗？** 在就做，不在才问
- 如果答案不确定，说明我没想清楚 → 先想清楚再决定问不问
- 问的时候不带"要不要"，只带"事实 + 我的结论 + 执行结果"

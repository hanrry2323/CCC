# Plan 文件格式规范

> ccc-planner 产出，Claude 消费。来源：planning-with-files task_plan.md + ccg-workflow spec-plan.md。

---

## 核心原则

Plan 是写给 Claude 的**自然语言指令**，不是脚本。

**验收口径（P3 修复）**：每条验收必须包含"自然语言意图"，可以附"参考命令"。

### 验收写法矩阵

| 写法 | 是否合规 | 说明 |
|------|---------|------|
| `python3 -c "import app.api"` | ❌ | 只有命令，没有意图 |
| "启动后端服务，确认健康检查返回正常" | ⚠️ 不充分 | 没有参考命令，Claude 可能选错工具 |
| "启动后端服务，确认健康检查返回正常（参考：`curl http://127.0.0.1:7777/api/health`）" | ✅ | 意图明确 + 参考命令 |
| "1. 跑 `git push origin main`" | ❌ | **步骤清单语气**，把 Claude 当 shell |
| "把本地 main 推到 origin，让外部 clone 拿得到（参考：`git push origin main`）" | ✅ | 目标意图 + 参考命令 |

**禁止**：
- ❌ 验收写"基本通过"/"差不多"/"看起来 OK"等模糊词
- ❌ 验收只有命令没有意图（违反红线 2）
- ❌ 验收只有意图没有参考命令（Claude 可能选错工具）
- ❌ 验收写"步骤 1 / 2 / 3"列表（这是命令清单，不是意图）

---

## "参考命令"使用边界（P3 强化 · 新增）

"参考命令"是 plan 写给 Claude 的**hint**，不是脚本步骤。规则：

| 规则 | ✅ 合规 | ❌ 违规 |
|---|---|---|
| **位置** | 放在意图句子末尾的括号里 | 单独成行、单独段落 |
| **句式** | "参考：`xxx`" 或 "如：`xxx`" | "步骤 1: `git push`" / "执行：`xxx`" |
| **数量** | 每条验收最多 1-2 个参考命令 | 一段验收 5 个命令堆叠 |
| **必填性** | 可选（hints），不强制 | 写成验收的唯一手段 |
| **具体度** | 给命令名即可（如 `git push`） | 给完整参数 + flags + pipe 链 |

**示例对比**：

```markdown
# ❌ 违规 — 步骤清单
### 验收
- 步骤 1：跑 `git push origin main`
- 步骤 2：跑 `git ls-remote origin refs/tags/v5.11.0`
- 步骤 3：跑 `git rev-list -1 v5.11.0`

# ✅ 合规 — 意图 + 参考
### 验收
- 本地 main 已推到 origin（参考：`git push origin main`）
- tag v5.11.0 在 origin 端可达（参考：`git ls-remote origin refs/tags/v5.11.0`）
- 外部 clone 后能拿到 tag 指向的 commit（参考：`git rev-list -1 v5.11.0`）
```

**为什么**：Claude 是自主 agent，让它根据 plan 意图自己决定命令、参数、调用顺序，比给步骤清单更可靠。步骤清单会让 Claude 跳过"判断"环节，变成纯执行器。

## 结构（三段式 + 必填字段）

```markdown
# Plan: [任务名]

## 范围
- 目标：一句话
- 只改文件：[列表]
- 不改文件：[列表]
- 执行方式：manual / auto / loop / goal（四选一）
- Phase 数：N（必填）

## 改动 N：[改什么]
### 做什么
### 怎么做
### 验收

## Commit 计划
| Phase | 改动 | Commit message 草稿 |
|-------|------|---------------------|

## 全局验收清单
- [ ] ...
```

---

## 执行方式选择（P2 修复）

| 方式 | 何时使用 |
|------|---------|
| `manual` | 一次性、单文件、可直接验证的小改（≤10 行） |
| `auto` | 简单多 phase、可逐步推进的改动 |
| `loop` | 需要定时轮询、反复执行（最长 7 天） |
| `goal` | 复杂多 phase、不中断跑完整个 plan |

**禁止其他术语**（如 "codeloop" / "手动" / "auto-loop"）。

---

## 约束

1. "验收"必须有自然语言意图 + 可选参考命令
2. "怎么做"必须具体到文件名 + 行号，禁止"优化结构"类描述
3. 一个改动仅对应一个文件修改，不要跨多个文件
4. 不准出现"后续考虑"、"待优化"等模糊项
5. 全局验收清单放在 plan 末尾，Verifier 终验时逐项确认
6. **每个 plan 都必须含 Commit 计划表**（P4 修复）
7. **每个 plan 都必须写 phases.json**（P1 修复：单 phase 改动至少 1 行 phase 1）
8. **plan 不写步骤清单**（P3 强化 · 新增）：用"自然语言意图 + 参考命令"取代步骤式描述
